# 🔴 Security Incident Runbook — Leaked Database Credential

**Severity:** Critical (P0) · **Type:** Hardcoded production-style secret committed to source control

## 1. Summary

A live MongoDB Atlas connection string — including username and password — was committed in
`CommBank-Server/Secrets.json` and loaded directly at startup by `Program.cs`. Anyone with read
access to the repository (or any fork, clone, or cached copy) can connect to the database with
full privileges.

| Item | Value |
| --- | --- |
| Exposed DB user | `rserver-user` |
| Cluster host | `rserver-cluster.60n9lra.mongodb.net` |
| Database | `rserver-db` |
| Exposure vector | `Secrets.json` tracked in git (no `.gitignore` rule) + loaded by `Program.cs` |

> **Assume the credential is compromised.** Removing the file from the latest commit is **not**
> sufficient — the secret remains in git history and in every clone/fork. The only reliable fix is
> to **rotate (invalidate) the credential** and then scrub history.

## 2. Immediate containment — do this now (minutes matter)

1. **Rotate the database password** (this instantly invalidates the leaked one):
   - MongoDB Atlas → **Database Access** → user `rserver-user` → **Edit** → **Edit Password** →
     **Autogenerate Secure Password** → **Update User**.
   - Or delete `rserver-user` entirely and create a fresh least-privilege user (see §5).
2. **Lock the network** while you investigate: Atlas → **Network Access** → remove any
   `0.0.0.0/0` (allow-from-anywhere) entry; allow only your app/runner egress IPs.
3. **Review the audit logs** for unauthorized access: Atlas → **Project → Activity Feed**, and
   (if on M10+) **Database Access History / Atlas auditing**. Look for unfamiliar source IPs or
   unexpected reads/writes since the file was first committed.
4. If any unauthorized access is found, treat as a **data breach** and trigger your
   incident-response / disclosure process.

## 3. Purge the secret from git history

The local working copy here has no `.git`, so these commands target the **origin** repository.
Coordinate with anyone who has clones — history rewrite is destructive and requires re-cloning.

**Option A — git filter-repo (recommended):**

```bash
# Work on a fresh mirror so you don't corrupt a working clone
git clone --mirror https://github.com/<owner>/<repo>.git
cd <repo>.git

# Remove the file from ALL history (adjust the path to match the repo layout)
git filter-repo --path CommBank-Server-main/CommBank-Server/Secrets.json --invert-paths

# Re-push the rewritten history
git push --force --all
git push --force --tags
```

**Option B — BFG Repo-Cleaner:**

```bash
bfg --delete-files Secrets.json
git reflog expire --expire=now --all && git gc --prune=now --aggressive
git push --force --all
```

After rewriting: ask all collaborators to **delete and re-clone**. On GitHub, also consider that
**forks and the pull-request ref cache may still hold the blob** — contact GitHub Support to purge
cached views if the repo is public, but rotation (§2) remains the real mitigation.

## 4. Re-secure the application (already applied in this branch)

The code now loads secrets from a layered, fail-fast provider chain instead of a committed file:

- `.gitignore` ignores `Secrets.json`, `*.local.json`, and `.env*`.
- `Secrets.json` has been scrubbed to a `REPLACE_ME` placeholder; `Secrets.example.json` is the
  committed template.
- `Program.cs` precedence (low → high): `appsettings*` → optional `Secrets.json` → user-secrets
  (Development) → **environment variables** (authoritative). It **throws at startup** if the
  connection string is missing or still a placeholder.
- `CommBank.csproj` has a `UserSecretsId` so `dotnet user-secrets` works locally.

**Local development setup (with the NEW rotated credential — never the leaked one):**

```bash
cd CommBank-Server-main/CommBank-Server
dotnet user-secrets set "ConnectionStrings:CommBank" "mongodb+srv://<new-user>:<new-pass>@<cluster-host>/<db>?retryWrites=true&w=majority"
```

**CI / Production:** provide the secret as the environment variable
`ConnectionStrings__CommBank` (double underscore), sourced from your secret manager
(Azure Key Vault, AWS Secrets Manager, Kubernetes Secret, GitHub Actions secret, etc.).
Do **not** ship `Secrets.json` in container images.

## 5. Least-privilege DB user (recommended)

Replace the broad user with a role scoped to only the app database:

- Atlas → **Database Access** → **Add New Database User**
- Auth: SCRAM, autogenerated password
- Privileges: **`readWrite`** on `rserver-db` only (no `atlasAdmin`, no cluster-wide roles)
- Restrict by **Network Access** allow-list to known egress IPs

## 6. Verification checklist

- [ ] Old password no longer works (test a connection with it → must fail)
- [ ] New secret provided only via env var / user-secrets / vault (not committed)
- [ ] `git log -p -- '**/Secrets.json'` on origin shows no credential after history rewrite
- [ ] `0.0.0.0/0` removed from Atlas Network Access
- [ ] App starts with the new secret and fails fast when it is absent
- [ ] Repo secret scanning enabled (GitHub **Secret Scanning** + **Push Protection**)

## 7. Prevention (next steps)

- Enable **GitHub Secret Scanning + Push Protection** on the repo.
- Add a pre-commit hook or CI step with a scanner (e.g. **gitleaks**) to block secrets.
- Add **Dependabot**/renovate; treat any future leaked secret as auto-rotate.
- Track follow-up hardening already identified: real authentication (login is currently a no-op),
  CORS lock-down, `net6`→`net8` LTS upgrade, and `[JsonIgnore]` on `User.Password`.
